# Django (Tier 3)

Balance policy: an existing consistent codebase pattern wins over textbook advice
unless it is a security, correctness, or performance defect.

## Rules

- **Follow the project's layering convention.** Fat models with thin views, or a
  dedicated service layer — pick whichever the codebase already uses and stay
  consistent. Views should orchestrate, not hold business rules.
- **Guard against N+1 queries.** Use `select_related` (FK/one-to-one, SQL join)
  and `prefetch_related` (M2M/reverse FK) whenever a loop touches related
  objects. Verify with `django-debug-toolbar` or `assertNumQueries`.
- **DRF serializers own API I/O.** Serializers validate and shape request and
  response data; keep query logic in the view/service, not the serializer.
- **Migration discipline.** Commit migrations with the model change, never edit
  an applied migration, and review autogenerated SQL for data-loss operations.
- **Split settings per environment.** `base` + `dev`/`prod`/`test` modules; read
  secrets from env (never commit them). Keep `DEBUG=False` guaranteed in prod.
- **Never trust request data.** Validate through a Form or serializer before use;
  never build ORM filters or raw SQL from unsanitized input.

## ❌ Bad — N+1 query

```python
# One query for authors, then one more per author for each book. O(n) queries.
for author in Author.objects.all():
    print(author.name, author.book_set.count())
```

## ✅ Good — prefetch the relation

```python
authors = Author.objects.prefetch_related("book_set")
for author in authors:  # Related books fetched in a single extra query.
    print(author.name, author.book_set.count())
```
